# Bug 001: CoW Path Handling Issues

> The entire generated bug report focus on file whose path are both non-whitelisted and non-blacklisted.

## Symptom

1. **Pre-existing file write fails silently**: Running `echo a > ../a.txt` inside the sandbox for a file that exists on the real FS (e.g., created with `touch ../a.txt` outside sandbox) appears to succeed but reading back returns nothing.

2. **New file creation fails**: Running `touch ../b.txt` inside the sandbox for a file that doesn't exist on the real FS fails with ENOENT.

## Possible bug points:

- `setattr(size=0)` for O_TRUNC
- `setattr` for `CopyOnWrite | FuseOnly`:
   - Checks `daemon.get_entry(path)` → returns `None` (not yet flushed)
   - Falls back to real FS via `real_path` → `fs::symlink_metadata` → ENOENT
   - **Replies `ENOENT`** ← BUG!
- relative/absolute path handling: we need a contract across modules for which one is passed

## Fix Specification

### Bug A: `setattr` ENOENT for new CopyOnWrite files

**Location**: `src/fuse/fs.rs:430-438` in the `setattr` method, `CopyOnWrite | FuseOnly` arm.

**Root Cause**: When `setattr` is called for a file that was just created via `create()` but not yet flushed to the daemon, `daemon.get_entry()` returns `None`. The fallback to real FS also fails (file doesn't exist there either), causing an incorrect `ENOENT` response.

**Fix**: When both daemon and real FS return nothing, synthesize a minimal `FileAttr` from the inode number. The file logically exists (it was just created via `create()`), so we should return success with a synthetic attr rather than ENOENT.

**Status**: IMPLEMENTED (see fs.rs:430-450)

---

### Bug B: Silent write data loss (echo appears to succeed but reading back returns nothing)

**Root Cause**: The **actual root cause** was in `src/syncing/disk.rs:init_sandbox()`:

The function creates `objects/` directory but **NOT** the subdirectories `objects/00/` through `objects/ff/` that are required by `ObjectStore::put()`. When `flush_to_daemon` tries to store a file via `objects.put(&data)`, the file creation fails because the parent directory doesn't exist. The error was silently ignored (flush returned success but data wasn't stored).

**Fix**: Added initialization of all 256 subdirectories (00-ff) in `init_sandbox()`:

```rust
// In src/syncing/disk.rs:init_sandbox()
let objects_dir = sandbox_data_dir.join("objects");
for i in 0..=0xff {
    let subdir = objects_dir.join(format!("{:02x}", i));
    fs::create_dir_all(&subdir)?;
}
```

**Status**: FIXED

---

### Bug C: CoW create denied on parent directories above project root

**Symptom**: Creating files via relative parent traversal (e.g. `touch ../x`) fails with `EACCES` only on specific ancestors of `pwd`.

**Root Cause**:

- For `CopyOnWrite`, when an entry is not present in daemon store, FUSE falls back to real filesystem metadata.
- In user namespace, many real owners/groups are unmapped and appear as overflow IDs (`uid/gid=65534` / `nobody`).
- Kernel performs parent-directory permission checks *before* sending `create`/`mknod` to FUSE.
- With fallback metadata (`owner!=caller`, mode often `0755`), write on parent fails at kernel side, so `create`/`mknod` is never invoked.

**Fix**:

- In `CopyOnWrite` mode, virtualize returned `FileAttr.uid/gid` to the mounting user/group for permission checks.
- Keep passthrough and fuse-only behavior unchanged.

This aligns permission checks with CoW semantics (writes go to fuse store, not host FS), while preserving strict host permissions in passthrough mode.

**Status**: IMPLEMENTED
